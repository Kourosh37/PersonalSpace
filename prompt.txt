Alternative if tus is not used:
Implement custom chunked upload:

================================================================================
MAX FILE SIZE ENFORCEMENT
================================================================================

Maximum upload file size must be configurable from the Admin Panel.

Default:
- Unlimited

Database settings:
- upload.max_file_size_mode = "unlimited"
- upload.max_file_size_bytes = null

If admin sets 10GB:
- upload.max_file_size_mode = "custom"
- upload.max_file_size_bytes = 10737418240

Rules:
- If upload.max_file_size_mode is "unlimited", do not enforce an app-level max file size.
- If upload.max_file_size_mode is "custom", enforce upload.max_file_size_bytes.

For normal streaming upload:
- If Content-Length is present and exceeds max, reject before reading body.
- If Content-Length is missing, count bytes while streaming and stop/reject when limit is exceeded.
- Return HTTP 413 Payload Too Large.

For tus/resumable uploads:
- Check Upload-Length at upload creation.
- If Upload-Length exceeds max, reject with 413.
- If Upload-Length is deferred/unknown, track received bytes and reject if it exceeds max.
- Prevent finalization of uploads larger than allowed size.
- Store rejected/failed state in upload_sessions.

Frontend:
- Fetch upload settings before upload.
- Show max file size in upload dialog.
- If a selected file exceeds max size, show a clear error before upload.
- Still rely on backend as the source of truth.
- If backend rejects due to size, show a polished error message:
  "This file exceeds the maximum allowed upload size."

Admin UI for max file size:
- Toggle/select:
  - Unlimited
  - Custom
- If custom:
  - numeric input
  - unit selector:
    - KB
    - MB
    - GB
    - TB
- Show computed bytes value
- Save button
- Reset to default button
- Show validation errors
- Show success toast after saving

DOWNLOAD REQUIREMENTS - VERY IMPORTANT
================================================================================

Downloading must be stable, streaming-based, and support resume.

Required download capabilities:
- Download any file type
- Download very large files
- Stream downloads
- Support HTTP Range requests
- Support browser resume capability
- Correct headers:
  - Content-Type
  - Content-Length
  - Content-Disposition
  - Accept-Ranges
  - ETag if possible
  - Last-Modified if possible
  - Cache-Control where appropriate

Private download endpoints must validate authentication.

Public shared download endpoints must validate share link permissions.

Range request behavior:
- Support Range: bytes=start-end
- Return 206 Partial Content when valid range is requested
- Return Content-Range
- Return Accept-Ranges: bytes
- Return 416 for invalid ranges
- This is required for:
  - large downloads
  - video streaming
  - audio streaming
  - PDF preview
  - browser download resume

Download progress:
Browser-native downloads do not always allow custom progress UI after handing off to the browser download manager.

Implement two download modes:

1. Browser native download:
- Use normal link/download endpoint
- Supports browser-managed resume if browser/server supports Range
- Best for huge files

2. In-app download progress where feasible:
- Use fetch/streams for files where browser allows it
- Show progress, speed, ETA
- Save using Blob/File System Access API where possible
- For very large files, prefer native browser download and explain limitations
- If using File System Access API, feature-detect and gracefully fallback

Important browser limitation:
If a native browser download starts and the user leaves the web page but browser remains open, the browser download manager may continue it.
If using in-app JavaScript fetch download, it may stop if tab is closed/suspended.
Document this clearly.

Folder download:
- Download folders as ZIP
- Stream ZIP response
- Do not load entire ZIP into RAM
- Preserve folder structure
- Support public shared folder ZIP download if allowed
- For ZIP streaming, Content-Length may not be known; that's acceptable
- Show UI state accordingly

Download UX:
- Show download button
- For files with known size, show size
- For in-app downloads, show:
  - percentage
  - speed
  - ETA
  - downloaded bytes / total bytes
  - pause/resume if technically supported
- For native downloads, show "Download started in browser"
- For failed downloads, show retry help

================================================================================
STORAGE DESIGN
================================================================================

Default storage:
- Local filesystem inside Docker volume.

Storage must be abstracted:

Storage interface should support:
- PutStream(ctx, key, reader)
- GetStream(ctx, key)
- GetRangeStream(ctx, key, start, end)
- Delete(ctx, key)
- Exists(ctx, key)
- Stat(ctx, key)
- Move/Copy if needed

Storage rules:
- Never trust user-provided paths.
- Never use raw file names as direct filesystem paths.
- Generate internal storage keys.
- Organize files by user/file ID or hashed path.
- Prevent path traversal.
- Use temporary paths for incomplete uploads.
- Atomically move/mark files complete after upload finalization.
- Keep metadata in database.

Example local storage layout:
/data/storage/files/...
/data/storage/tmp/uploads/...
/data/storage/previews/...

Optional future storage:
- MinIO/S3 through same interface.
